# Security Policy

## الإصدارات المدعومة

يتم دعم الإصدارات التالية من Arabic HTTP Studio بالتحديثات الأمنية:

| الإصدار | مدعوم |
|---------|-------|
| 1.0.x   | ✅    |
| < 1.0   | ❌    |

## الإبلاغ عن ثغرة أمنية

نأخذ الثغرات الأمنية على محمل الجد. شكرًا لتحسين أمن Arabic HTTP Studio.

**يرجى عدم الإبلاغ عن الثغرات الأمنية عبر GitHub issues العامة.**

بدلاً من ذلك، يرجى الإبلاغ عنها بالبريد الإلكتروني: **security@arabic-http-studio.com**

### ما يجب تضمينه في التقرير

- وصف الثغرة ونوعها
- خطوات إعادة الإنتاج
- تأثير الثغرة
- الإصدار المتأثر
- أي معلومات إضافية مفيدة

### وقت الاستجابة

- **التأكيد**: خلال 48 ساعة
- **التقييم الأولي**: خلال 7 أيام
- **الإصلاح**: حسب خطورة الثغرة
  - حرجة: خلال 7 أيام
  - عالية: خلال 30 يومًا
  - متوسطة: خلال 60 يومًا
  - منخفضة: في الإصدار التالي

## ممارسات الأمان

### التخزين الآمن

- نستخدم `flutter_secure_storage` لتخزين البيانات الحساسة
- المفاتيح المشفرة مخزنة في Android Keystore / iOS Keychain
- لا نخزن كلمات المرور في نص عادي

### التشفير

- نستخدم AES-256 للتشفير
- يتم إنشاء مفاتيح التشفير بشكل عشوائي
- لا نستخدم خوارزميات تشفير ضعيفة

### الشبكة

- ندعم TLS 1.2+ افتراضيًا
- يمكن تكوين Certificate Pinning
- لا نسمح بـ HTTP غير المشفر افتراضيًا

### المصادقة

- ندعم مصادقة Bearer Token
- ندعم Basic Auth
- ندعم Digest Auth
- ندعم API Keys
- لا نخزن الرموز في سجلات التطبيق

## مكافآت الأمان

نحن نقدر الإبلاغ المسؤول عن الثغرات الأمنية. قد نقدم:

- التقدير العلني (بإذنك)
- إدراج اسمك في قائمة المساهمين الأمنيين
- مكافآت رمزية للثغرات الحرجة

## التحديثات الأمنية

سيتم الإعلان عن التحديثات الأمنية عبر:

- GitHub Security Advisories
- صفحة الإصدارات
- البريد الإلكتروني للمستخدمين المسجلين

## سياسة الإفصاح

نتبع سياسة الإفصاح المنسق:

1. استلام التقرير
2. التأكيد والتقييم
3. العمل على الإصلاح
4. الإصدار والإفصاح العام (بعد 90 يومًا أو بعد الإصلاح)
